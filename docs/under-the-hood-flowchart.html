Architecture Walkthrough

FullVision Under the Hood

This page explains the app as one main story: a teacher opens FullVision, works inside a desktop or mobile shell, the app writes to a local cache immediately, and only then syncs into Supabase’s canonical backend. Everything else is a supporting detail around that loop.

User Flowcharts

These are the teacher-facing journeys. Read these first if the goal is to understand actual use of the product.

First-Time Use: Login To Printed Reports

The shortest realistic first-run path for a teacher who wants to get to reporting.

1
Start

Log in and land on the teacher dashboard

The teacher signs in, enters the teacher interface, and sees the desktop dashboard as the home base.

2
Class Setup

Create or import the class roster

The teacher confirms the active course, adds students manually or imports a roster, and makes sure the class list is usable.

3
Assessment Setup

Create the first assignment and connect it to learning targets

The teacher creates an assignment, picks the assessment type, and links it to the right tags or sections.

4
Evidence Entry

Enter scores, notes, or observations for students

The teacher begins building evidence by grading work, marking proficiency, and recording comments or observations.

5
Reporting

Open the reports page and build the report view

The teacher goes to reports, chooses the report structure, reviews narratives, and checks that each student’s report looks right.

6
Output

Print or export the finished reports

The teacher produces the final parent-facing output after reviewing the report content and layout.

Teacher Workflow: Assignments, Comments, And Grades

The recurring classroom workflow for building assessments and recording evidence.

1
Planning

Open the assignments area for the current course

The teacher enters the assignments workflow for the active course and reviews existing assessment items.

2
Assignment Setup

Create or edit the assignment details

The teacher adds the title, date, type, tags, rubric, and any weight or scoring mode for the task.

3
Grading

Enter grades or proficiency for each student

The teacher records grades in the assignment view, gradebook, or mobile grading surface depending on context.

4
Comments

Add comments, observations, or feedback

The teacher records comment-level evidence that gives context beyond a score and helps with later reporting.

5
Review

Review progress in gradebook, dashboard, or student view

The teacher checks whether the evidence looks complete, which students need attention, and how proficiency is trending.

6
Reuse

Carry that evidence into reports

The same assignments, grades, and comments later feed the reporting workflow without re-entry.

Main Flow

The shortest correct mental model of the whole system. If this part makes sense, the app makes sense.

1
Entry + Delivery

Teacher opens the app

Netlify serves HTML, injects environment values, and the browser decides whether the teacher lands in the login page, desktop SPA, or mobile PWA shell.

inject-env.js login-auth.js teacher/app.html teacher-mobile/index.html
2
UI Runtime

The shell takes over

Desktop uses a router and page modules. Mobile uses a tab shell and touch-first navigation. They look different, but they are two front ends over the same shared model.

teacher/router.js teacher-mobile/shell.js
3
Local-First State

The app reads and writes locally first

shared/data.js is the center of gravity. Screens read synchronously from memory and localStorage. When a teacher changes data, the app updates locally immediately so the interface stays fast and resilient.

shared/data.js docs/ARCHITECTURE.md
4
Interpretation Layer

The app turns evidence into meaning

Raw scores, statuses, overrides, and observations are interpreted by the calculation and reporting layers into proficiency, dashboards, and parent-facing narratives.

shared/calc.js teacher/page-reports.js report-blocks.js
5
Cloud Canon

Background sync updates the official record

Local writes are pushed to Supabase through guarded RPCs. The canonical multi-schema database is the official source of truth, but the teacher never has to wait for it during normal interaction.

shared/supabase.js schema.sql

Desktop vs Mobile

Same facts underneath. Different interaction surfaces on top.

Desktop

Best thought of as a classic single-page app. A hash router swaps page modules like dashboard, assignments, gradebook, observations, and reports.

  • Navigation model: router + page lifecycle
  • Best for: broad class management and dense workflows
  • Primary files: teacher/app.html, teacher/router.js, page modules

Mobile

Best thought of as a compact PWA shell. It uses persistent tabs, touch gestures, pull-to-refresh, and quick capture patterns for in-the-moment classroom work.

  • Navigation model: tab shell + stacked subviews
  • Best for: observation, quick grading, hallway use
  • Primary files: teacher-mobile/index.html, teacher-mobile/shell.js

The Sync Loop

This is the most important technical trick in the app.

1. Teacher edits data

A score, note, student record, or report setting changes.

2. Local state updates immediately

Memory and localStorage are updated right away so the UI remains responsive.

3. Background RPC runs

The app sends a canonical write to Supabase without blocking the teacher.

4. Canonical IDs and truth settle

The backend returns official IDs and durable state, which the local cache patches back in.

Optional Deep Dives

Only open these after the main flow feels obvious.

Authentication and session behavior

FullVision uses Supabase Auth, but the client tries a fast local token path before calling the network. Sign-out also clears local app state for shared-computer safety.

Why the data layer matters more than the UI

The page modules are comparatively straightforward. The real architecture lives in shared/data.js, where caching, retries, inflight tracking, and cross-tab coordination live.

What the calculation engine actually does

shared/calc.js is not storage. It is interpretation. It transforms raw evidence into tag, section, and overall proficiency using the chosen pedagogy and weighting rules.

What “canonical” means here

The canonical backend is the official database representation in Supabase. The frontend’s local state is optimized for speed and resilience, but the RPC-backed multi-schema database is the durable record.
